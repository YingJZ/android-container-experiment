# 调度优化收益预估方案

## 评估范围

只统计这类损失：

- 容器内关键线程已经 runnable
- 但由于容器前台/启动 boost 没及时生效，仍按普通线程待遇参与宿主机调度
- 导致它比“应有状态”晚拿到 CPU

只要出现下面任一情况，就**不计入本次收益**：

- 线程在等 I/O、page fault、direct reclaim、futex/锁
- 线程已经拿到目标优先级，但仍因总 CPU 不够而等待
- 跟首屏关键路径无关
- 是解冻、dex2oat/JIT、下载、业务逻辑本身耗时

## 评估计划

### 关键线程

- App 主线程
- RenderThread
- App Binder 线程中真正参与首屏链路的线程
- 容器内 Android 系统侧 system_server 上对应的服务线程
- 图形/窗口链路里真正阻塞首帧的线程

### 收益计算的具体标准
只有满足下面 4 条，才算到收益里：

- 线程在关键路径上
- 线程状态是 runnable，不是 blocked
- 此时它还没进入“应有”的前台/启动 boost 状态。
    - 具体来说，被优先调度的进程状态应该符合：（参考 [判断容器内进程是否进入目标优先级的具体方法.md](./判断容器内进程是否进入目标优先级的具体方法.md) ）
        - 调度切片（sched）的 `priority` 字段 <=110
        - cgroup 进入 `/top-app` 或 `/foreground`
        - Perfetto 的 `oom_score_adj_update` 事件 (辅助信号，用于确认进程是否
        - CPU affinity / cpuset 是否位于大核
    

### 具体评估方案：分阶段分析 gate thread

每个阶段只找“当前 gate thread”，避免重复计数。

1. T0 -> start proc 完成
    - 看容器内 system_server、zygote、app 主线程
    - 关注：启动触发后，相关线程多久才真正进入“前台/启动态”
2. bindApplication -> Activity.onCreate
    - 看 app 主线程、Binder 对端线程
    - 关注：同步 Binder 往返里，有没有明显的 runnable delay
3. onCreate -> first traversal / first draw
    - 看 app 主线程、RenderThread
    - 关注：主线程和渲染线程是否被宿主机其他线程频繁插队
4. first draw -> fully drawn
    - 只看仍在首屏关键路径上的线程
    - 业务异步加载、网络下载不要算进来

| 阶段       | Gate Thread        | Runnable Delay(ms) | 是否计入预估收益                 |
| :--------- | :----------------- | :----------------- | :------------------------------- |
| 启动拉起   | system_server 线程 |                    | 是                               |
| 进程初始化 | App Main           |                    | 是                               |
| 首帧构建   | App Main           |                    | 视情况，如果有 page fault 就排除 |
| 首帧渲染   | RenderThread       |                    | 是                               |
| 首屏完成   | 某 Binder 对端     |                    | 视情况，网络/异步加载排除        |

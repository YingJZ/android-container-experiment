好问题！在 Perfetto 里判断"容器内进程/线程是否进入目标优先级"是整个分析的核心，但也是最容易搞混的地方。下面给你一份**可直接操作的判定方法**。

---

## 一、你要看的信号源

在 Perfetto 里，优先级信息藏在这几个地方：

### 1. **调度切片的 `priority` 字段**
最直接的信号。

**怎么看：**
- 选中任意一个线程的 `sched` 切片
- 右侧 `Details` 面板会显示：
  ```
  priority: 120  // 或其他数值
  ```

**含义：**
- Linux 调度优先级范围：`0~139`
  - `0~99`：实时优先级（RT）
  - `100~139`：普通优先级（CFS），对应 nice `-20~19`
  - `120` = nice `0`（默认）
  - `110` = nice `-10`（前台）
  - `100` = nice `-20`（最高普通优先级）

**目标状态：**
- **容器前台/启动 boost** 通常应该让关键线程进入：
  - `priority ≤ 110`（nice ≤ -10）
  - 或者进入 RT 范围（`priority < 100`）

**判定方法：**
- 如果你看到 App Main / RenderThread 在启动窗口内 `priority` 仍然是 `120` 或更高（数值越大优先级越低），说明**没及时提权**。

---

### 2. **cgroup 路径**
第二重要的信号。

**怎么看：**
- 选中线程切片
- 在 `Details` 面板找 `cgroup` 字段，或者在 SQL 里查：
  ```sql
  SELECT ts, dur, utid, thread.name, 
         sched.ucpu, sched.priority,
         process.cgroup_path
  FROM sched
  JOIN thread USING(utid)
  JOIN process USING(upid)
  WHERE thread.name = 'your_app_main_thread'
  ```

**典型路径：**
- `/background` → 后台
- `/foreground` → 前台
- `/top-app` → 顶层应用
- `/system-background` → 系统后台
- `/restricted` → 受限

**目标状态：**
- 容器内关键进程在启动窗口应该进入：
  - `/top-app` 或
  - `/foreground`

**判定方法：**
- 如果在 `T0 -> T1` 窗口内，App 进程仍在 `/background` 或 `/system-background`，说明**cgroup 迁移不及时**。

---

### 3. **OOM adj**
辅助信号，用于确认进程是否被识别为前台。

**怎么看：**
- Perfetto 的 `oom_score_adj_update` 事件
- 或在 SQL 里查：
  ```sql
  SELECT ts, track_id, int_value AS oom_adj
  FROM counter
  JOIN process_counter_track ON counter.track_id = process_counter_track.id
  WHERE process_counter_track.name = 'oom_score_adj'
    AND process_counter_track.upid = (
      SELECT upid FROM process WHERE name = 'your.app.package'
    )
  ```

**典型值：**
- `0` → 前台应用
- `100` → 可见应用
- `200` → 感知服务
- `≥900` → 缓存/后台

**目标状态：**
- 启动窗口内应该是 `0` 或 `100`

**判定方法：**
- 如果 `oom_adj` 在 `T0` 之后很久才从 `900+` 降到 `0`，说明**AMS 状态更新滞后**。

---

### 4. **CPU affinity / cpuset**
容器场景特有的坑。

**怎么看：**
- Perfetto 没有直接显示，需要在 SQL 里查或结合 `sched_switch` 事件推断：
  ```sql
  SELECT ts, utid, cpu, thread.name
  FROM sched
  JOIN thread USING(utid)
  WHERE thread.name = 'your_thread'
  ORDER BY ts
  ```

**典型问题：**
- 容器内进程被限制在 `cpuset.background`（通常只有小核）
- 即使 `priority` 提升了,仍然无法跑到大核

**判定方法：**
- 看关键线程是否只在 `cpu 0-3`（小核）上跑
- 如果启动窗口内从未出现在 `cpu 4-7`（大核），且此时大核空闲,说明**cpuset 迁移不及时**。

## 二、实操：在 Perfetto 里逐步判定

### **Step 1：先找到目标线程**
1. 打开 Perfetto
2. 按 `/` 搜索线程名，比如：
   - `your.app.package`（主进程）
   - `RenderThread`
   - `Binder:` 开头的线程
3. 固定这几个线程到视图顶部（右键 → Pin to top）

---

### **Step 2：标出时间窗口**
1. 找到 `T0`（启动触发点）
   - 搜索 `am_start` 或 `activityStart`
   - 或看 `system_server` 里的 `startActivity` slice
2. 找到 `T1`（首帧）
   - 搜索 `Choreographer#doFrame` 或 `DrawFrame`
3. 找到 `T2`（fully drawn）
   - 搜索 `reportFullyDrawn` 或自定义埋点

在时间轴上用 `m` 键打标记（marker）。

---

### **Step 3：逐段检查优先级**
选中 App Main 的 `sched` 切片，从 `T0` 开始往后看：

| 时间点   | priority | cgroup      | oom_adj |  CPU | 判定            |
| -------- | -------: | ----------- | ------: | ---: | --------------- |
| T0       |      120 | /background |     900 | cpu0 | ❌ 未提权        |
| T0+200ms |      120 | /background |     900 | cpu1 | ❌ 未提权        |
| T0+500ms |      110 | /foreground |       0 | cpu2 | ✅ 已提权        |
| T0+800ms |      110 | /top-app    |       0 | cpu6 | ✅ 已提权 + 大核 |

**关键判定点：**
- 如果 `T0 -> T0+500ms` 这 500ms 内，线程一直是 `priority=120` + `/background`，说明**提权延迟了 500ms**。

---

### **Step 4：确认是否真的造成损失**
光看到"没提权"还不够，还要证明"因此它被延迟了"。

**方法：**
1. 在那 500ms 窗口内，找到 App Main 的 `runnable` 空洞
   - 就是 `sched` 切片之间的空白
2. 点击空洞，看 `Details` 面板的 `state`：
   - 如果是 `R`（Runnable），说明它在等 CPU
   - 如果是 `S`（Sleep）/ `D`（Disk Sleep），不算
3. 看同时刻其他 CPU 在跑什么：
   - 如果在跑 `priority=120` 或更低的宿主机线程，说明**容器线程被插队了**
   - 如果所有 CPU 都在跑更高优先级任务，说明"即使提权也拿不到 CPU"，不算

**只有同时满足：**
- 线程是 `R`（Runnable）
- 它还没提权
- 宿主机 CPU 在跑更低优先级任务

**才算到 `ContainerPriorityPenalty`。**

---

## 三、SQL 一键查询版

如果你要批量分析，直接用 SQL：

```sql
-- 1. 找到目标进程的 upid
WITH target_process AS (
  SELECT upid FROM process WHERE name = 'your.app.package'
),

-- 2. 查询启动窗口内的优先级变化
priority_timeline AS (
  SELECT 
    ts,
    dur,
    utid,
    thread.name AS thread_name,
    sched.priority,
    sched.end_state,
    process.cgroup_path
  FROM sched
  JOIN thread USING(utid)
  JOIN process USING(upid)
  WHERE process.upid IN (SELECT upid FROM target_process)
    AND ts BETWEEN <T0_ns> AND <T2_ns>  -- 替换成你的时间窗口
  ORDER BY ts
),

-- 3. 找出"未提权"的 runnable 时段
penalty_windows AS (
  SELECT 
    ts,
    dur,
    thread_name,
    priority,
    cgroup_path
  FROM priority_timeline
  WHERE priority > 110  -- 认为 >110 就是"未提权"
    AND end_state = 'R'  -- 只看 runnable
)

SELECT 
  thread_name,
  SUM(dur) / 1e6 AS total_penalty_ms,
  COUNT(*) AS penalty_count
FROM penalty_windows
GROUP BY thread_name
ORDER BY total_penalty_ms DESC;
```

**输出示例：**
```
thread_name          | total_penalty_ms | penalty_count
---------------------|------------------|---------------
your.app.package     | 320.5            | 18
RenderThread         | 85.2             | 7
Binder:12345_2       | 42.1             | 3
```

## 四、常见坑与排除项

### ❌ **坑 1：把所有 runnable delay 都算进来**
**错误做法：**
> "App Main 有 500ms runnable delay，所以收益就是 500ms。"

**正确做法：**
- 只算"未提权 + runnable + 宿主机在跑更低优先级任务"的部分
- 如果它已经是 `priority=110` 但仍在等，说明是 CPU 总量不够，不是优先级问题

---

### ❌ **坑 2：忽略 cgroup 和 cpuset**
**典型现象：**
- `priority` 已经是 `110`
- 但线程仍然只在小核上跑
- 大核空闲

**原因：**
- `cgroup` 还在 `/background`，导致 `cpuset` 限制没解除

**判定：**
- 这也算"优先级调度不及时"，因为 cgroup 迁移是优先级生效的一部分

---

### ❌ **坑 3：把 I/O 等待算进来**
**典型现象：**
- 线程 `state = D`（Disk Sleep）
- 或在等 `page fault`

**判定：**
- 这不是调度问题，是 I/O 问题，**不计入**

---

### ❌ **坑 4：把业务逻辑耗时算进来**
**典型现象：**
- 线程在 `S` 状态，等待网络回包
- 或在 `onCreate` 里跑了 200ms 的同步初始化

**判定：**
- 这是业务问题，不是调度问题，**不计入**

---

## 五、最终判定清单

每次看到一个空洞，按这个顺序检查：

| 检查项                      | 怎么看                                   | 通过条件                                 |
| --------------------------- | ---------------------------------------- | ---------------------------------------- |
| 1. 是关键路径线程？         | 看线程名 + 时间段                        | 是 App Main / RenderThread / 关键 Binder |
| 2. 是 Runnable？            | `end_state = R`                          | 是                                       |
| 3. 未提权？                 | `priority > 110` 或 `cgroup=/background` | 是                                       |
| 4. 宿主机在跑低优先级任务？ | 看其他 CPU 的 `sched`                    | 是                                       |
| 5. 不是 I/O/锁？            | `state ≠ D/S`, 无 `page fault`           | 是                                       |

**只有 5 个全过，才计入 `ContainerPriorityPenalty`。**

## 七、Shadow Boost 怎么验证

如果你做了 Shadow Boost 实验，判定就简单了：

**对比两条 trace：**
1. Baseline：App Main 在 `T0~T0+500ms` 内 `priority=120`
2. Shadow Boost：同时段 `priority=110`

**看差异：**
- Baseline 的 runnable delay：320ms
- Shadow Boost 的 runnable delay：50ms
- **差值 270ms 就是可归因收益**

这比手工判定可靠得多。

## 八、总结：你现在就能做的

1. **打开一条 baseline trace**
2. **找到 App Main，标出 T0 和 T1**
3. **从 T0 开始，每隔 100ms 点一个 sched 切片，记录：**
   - `priority`
   - `cgroup_path`
   - `oom_adj`（如果有）
4. **找到第一个 `priority ≤ 110` 的时间点**
5. **计算 `T0` 到这个时间点的差值 = 提权延迟**
6. **在这段延迟内，统计 runnable 时长**
7. **排除 I/O、锁、业务逻辑**
8. **剩下的就是 `ContainerPriorityPenalty`**

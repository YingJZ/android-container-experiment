# 「快照恢复 + AMS 伪装冷启动」方案：Binder 修复后的残余问题

> 本文档评估一种替代方案的可行性：CRIU restore 后，不尝试精确恢复 AMS 侧组件状态，而是让进程"伪装成新启动"——调用 attachApplication 注册、跳过 bindApplication 业务初始化、主动调用 startActivity 获取全新 Activity。
>
> 前提条件：Binder 内核态已完整恢复（handle 编号、node/ref/thread/death 全部重建），app 进程可以正常发起 Binder IPC。

## 一、方案概述

```
阶段 1：进程复活与伪装
  CRIU restore → 进程恢复，内存中有旧 Application/Activity/ClassLoader 等
  调用 AMS.attachApplication(mAppThread) → AMS 为新 PID 创建 ProcessRecord

阶段 2：应付 AMS 的 bindApplication
  AMS 发送 bindApplication → app 侧接收
  执行：更新 Configuration、系统服务引用
  跳过：makeApplication / Application.onCreate / installContentProviders

阶段 3：主动申请启动
  调用 AMS.startActivity(ACTION_MAIN + CATEGORY_LAUNCHER + FLAG_ACTIVITY_NEW_TASK)
  AMS 创建新 ActivityRecord + 新 Token
  AMS 下发 scheduleLaunchActivity

阶段 4：新 Activity 诞生
  Instrumentation.newActivity() 反射创建新 Activity
  activity.attach(新 Token)
  onCreate → onStart → onResume
```

## 二、致命问题

### 2.1 PID 未在 mPidsSelfLocked 中注册 → 进程被杀

**现象**：CRIU restore 后 PID 变化（或保持不变但 AMS 认为旧进程已死），`attachApplicationLocked()` 第一步通过 PID 查找 ProcessRecord：

```java
// AMS.java L4284-L4341
app = mPidsSelfLocked.get(pid);
// ...
if (app == null && startSeq > 0) {
    app = mProcessList.mPendingStarts.get(startSeq);
}
if (app == null) {
    // ⛔ 没找到 → 直接杀进程
    killProcessQuiet(pid);
    return false;
}
```

**问题本质**：`attachApplication` 的设计假设是"AMS 已经通过 `startProcessLocked` 为该 PID 创建了 ProcessRecord"。一个凭空出现的 PID 不在 AMS 的任何注册表中。

**影响等级**：致命。进程直接被杀，后续所有步骤无法执行。

**可能的解法**：

| 解法 | 可行性 | 代价 |
|:-----|:------:|:-----|
| 保持 PID 不变 + AMS 侧旧 ProcessRecord 仍在 | 低 | CRIU 必须保证 PID 一致；AMS 可能在 restore 期间已经清理了旧 ProcessRecord |
| 在 AMS 中预先为新 PID 注入 ProcessRecord | 中 | 需要新增 AMS 接口（如 `prepareRestoreProcess`），修改 `mPidsSelfLocked` / `mProcessNames` / `mActiveUids` |
| 复用 `startProcessLocked` 路径让 AMS 创建 ProcessRecord | 中 | 需要让 AMS 走完"准备启动新进程"流程但不实际 fork Zygote，需要 hook Zygote 连接 |
| 利用 `mPendingStarts` 路径 | 低 | 需要构造合法的 `startSeq`，且 startSeq 与 `startProcessLocked` 的内部计数器关联 |

**推荐方向**：新增 `prepareRestoreProcess(pid, uid, processName)` 接口，在 CRIU restore 前由容器管理侧调用，让 AMS 为即将恢复的进程预先创建 ProcessRecord。这等同于 PLAN.md 的 AMS-Reconnection.md 中建议的 `reattachApplicationLocked` 的前置步骤。

### 2.2 bindApplication 不可简单跳过

**现象**：AMS **无条件发送** bindApplication，且客户端不执行则后续全部崩溃。

```java
// AMS.java L4487-L4523 — 两个分支都发送 bindApplication，无跳过路径
thread.bindApplication(processName, appInfo, providerList, ...);
```

客户端 `handleBindApplication` 中的关键步骤及其依赖关系：

```
mBoundApplication = data                    ← 后续所有代码依赖此字段
├── 设置 Configuration                      ← 资源加载正确性
├── ResourcesManager.applyConfiguration     ← 资源匹配
├── getPackageInfoNoCheck()                 ← ClassLoader
├── 创建 Instrumentation                    ← Activity 创建的前置条件
├── ContextImpl.createAppContext()          ← Context
├── NetworkSecurityConfigProvider.install() ← 网络安全
├── makeApplication()                       ← 有幂等保护，可有限跳过
├── installContentProviders()               ← ⛔ 10 秒超时→杀进程
└── callApplicationOnCreate()               ← 可跳过但有风险
```

**子问题 A：ContentProvider 发布超时**

AMS 在发送 bindApplication 后会设置 10 秒超时：

```java
// AMS.java L4395-L4399
Message msg = mHandler.obtainMessage(CONTENT_PROVIDER_PUBLISH_TIMEOUT_MSG);
msg.obj = r;
mHandler.sendMessageDelayed(msg, CONTENT_PROVIDER_PUBLISH_TIMEOUT);
```

如果 app 不调用 `publishContentProviders`，10 秒后 AMS 杀掉进程。

**子问题 B：makeApplication 的幂等保护是双刃剑**

```java
// LoadedApk.java L1318-L1319
if (mApplication != null) {
    return mApplication;  // 快照恢复后 mApplication 已存在，直接返回
}
```

快照中 `mApplication != null`，`makeApplication` 会跳过创建。但此时：
- 新的 `ContextImpl` 已在 `handleBindApplication` 前面被创建
- 新的 `Instrumentation` 已被创建
- 旧 Application 对象的 `mBase`（ContextWrapper 的 base context）指向快照中的旧 ContextImpl → 可能与新 Configuration 不一致

**子问题 C：Instrumentation 替换导致 Activity 生命周期异常**

`handleBindApplication` 会创建新的 `Instrumentation` 实例。如果旧 Activity 是由旧 Instrumentation 创建的，而新 Activity 由新 Instrumentation 创建，两者对 `onCreate` / `onResume` 等回调的处理可能不一致。

**影响等级**：致命。不处理 ContentProvider 发布 → 进程被杀；不初始化 Instrumentation/Context → Activity 无法正常创建。

**可能的解法**：

| 解法 | 可行性 | 代价 |
|:-----|:------:|:-----|
| 执行完整 bindApplication | 低 | 等于没省时间，与冷启动无异 |
| 选择性执行 + 手动 publishContentProviders | 中 | 需要在 framework 层增加"restore mode"标志，修改 handleBindApplication 逻辑；用快照中已有的 CP 对象手动构造 ContentProviderHolder 列表 |
| 新增 reattachApplicationLocked 接口 | 中 | PLAN.md 方案已设计此接口——执行步骤 4-6, 8-10 跳过步骤 3, 7 |

**推荐方向**：采用 `reattachApplicationLocked`（AMS-Reconnection.md 已设计），不走标准 bindApplication 路径。如果必须走 bindApplication，则需要增加"restore mode"标志位，让 `handleBindApplication` 区分正常启动和快照恢复。

### 2.3 scheduleLaunchActivity 始终创建新 Activity → 旧 Activity 变幽灵

**现象**：AOSP 中 `scheduleLaunchActivity` 的处理链路始终通过反射创建全新 Activity 实例：

```java
// ActivityThread.java L3533-L3537
activity = mInstrumentation.newActivity(cl, component.getClassName(), r.intent);
// ← 始终创建新实例，不存在"复用已有 Activity"的路径
```

同时，`LaunchActivityItem.preExecute` 始终创建新的 `ActivityClientRecord`：

```java
// LaunchActivityItem.java L84-L105
ActivityClientRecord r = new ActivityClientRecord(token, mIntent, ...);
client.addLaunchingActivity(token, r);
```

**问题本质**：主动调用 `startActivity` 会产生一个全新 Activity 实例 + 新 Token。快照中旧的 Activity 实例变成"幽灵"——它被大量内部对象引用，无法安全丢弃。

**旧 Activity 的悬挂引用清单**：

| 引用来源 | 引用方式 | 丢弃后果 |
|:---------|:---------|:---------|
| ViewModelStore | `Activity.mViewModelStore` → 被 Fragment/Adapter 使用 | 数据丢失、NPE |
| Handler 回调 | 匿名内部类持有 `this` 引用 | 回调指向已死 Activity → NPE |
| 异步任务 | 网络请求/数据库查询的回调 | 结果无法投递或投递到死 Activity |
| View 树 | `DecorView` → `Window` → `WindowManagerImpl` | 内存泄漏（View 引用 Activity Context） |
| Loader / AsyncTask | 旧 Activity 作为 Context | 资源泄漏 |
| 静态变量 | SDK 单例引用旧 Activity | 后续 SDK 调用可能操作死 Activity |

**影响等级**：致命。不丢弃 → 两套 Activity 实例共存导致状态冲突；丢弃 → 上述全部悬挂引用导致崩溃。

**可能的解法**：

| 解法 | 可行性 | 代价 |
|:-----|:------:|:-----|
| 不创建新 Activity，而是恢复旧 Activity 的 Token | 高 | 这正是 PLAN.md 方案的做法——重新对齐 ActivityRecord 与 app 侧 Activity 实例，而非创建新的 |
| 创建新 Activity + 手动 destroy 旧 Activity | 低 | `ActivityThread.performDestroyActivity` 不会清理所有悬挂引用；SDK/异步任务不可控 |
| 创建新 Activity + 让旧 Activity 走 finish 路径 | 低 | 用户可能看到闪烁；旧 Activity 的 Window 可能与新 Activity 冲突 |

**推荐方向**：**不主动调 startActivity**。让 AMS 按 ActivityRecord 当前状态重新调度，重新对齐 app 侧 Activity 与 AMS 侧 ActivityRecord。这正是 PLAN.md AMS-Reconnection Step 3 的做法。

### 2.4 旧 Binder 死亡通知触发级联清理

**现象**：即使 Binder 已修复，CRIU restore 过程中仍存在一个时间窗口——旧 `IApplicationThread` Binder 代理的死亡通知可能在 `mThread` 更新之前到达 AMS：

```
时间线：
  T0: CRIU restore 完成，进程恢复
  T1: 旧 IApplicationThread Binder 代理死亡 → 内核发送 BR_DEAD_BINDER
  T2: AMS 的 AppDeathRecipient.binderDied() 被触发
  T3: appDiedLocked() → handleAppDiedLocked() → cleanUpApplicationRecordLocked()
      → 从 mPidsSelfLocked 移除 → makeInactive → Kill 所有 Service → 通知 ATMS 清理 Activity
  T4: 你的代码尝试 attachApplication → ProcessRecord 已被清理 → 失败
```

**关键代码**：

```java
// AMS.java L3399-L3486 — appDiedLocked 的身份校验
if (app.getPid() == pid && app.getThread().asBinder() == thread.asBinder()) {
    handleAppDiedLocked(app, pid, false, true, fromBinderDied);
}
```

**影响等级**：致命。级联清理会摧毁 ProcessRecord，后续所有 AMS 交互不可能成功。

**可能的解法**：

| 解法 | 可行性 | 代价 |
|:-----|:------:|:-----|
| 在旧 Binder 代理死亡通知到达前完成 mThread 更新 | 中 | 需要精确控制时序；restore 后的第一时间就更新 ProcessRecord.mThread |
| 先 unlinkToDeath 旧 DeathRecipient | 中 | 需要在 AMS 侧执行，要求 restore 编排器能在 AMS 进程中操作 |
| 让 AMS 进入"restore 模式"，暂时忽略死亡通知 | 中 | 需要 AMS 侧修改，增加 restore 状态标记 |

**推荐方向**：在 AMS 侧增加 `markProcessRestoring(pid)` 接口，restore 前调用，让 AMS 暂时忽略该进程的死亡通知。restore 完成后调用 `finishProcessRestore(pid)` 恢复正常监控。这与 PLAN.md 建议的"标记 ProcessRecord 为 restoring 状态"一致。

## 三、重大缺陷

### 3.1 WMS / Surface / Input 重建成本未被节省

方案在"阶段 4：新 Activity 的诞生"后就结束了。但新 Activity 要显示到屏幕上，仍需完整走完 WMS 链路：

```
WMS addView / relayoutWindow
  → WindowState 创建
  → SurfaceControl 创建
  → BufferQueue 分配
  → InputChannel 建立与注册
  → Choreographer 获取 vsync
  → RenderThread 绑定 Surface
  → 首帧 measure/layout/draw
  → SurfaceFlinger 合成显示
```

这与正常冷启动的 WMS 成本**几乎相同**。方案省的主要是 Application 初始化时间（ClassLoader + ContentProvider + Application.onCreate），但首帧渲染链路仍然要走完。

**量化影响**：在 PLAN.md 的甜点区（onResume 完成 + addView 前）做快照时，WMS 链路是冷启动中相对轻的部分（~50-100ms），但如果你的 Activity 恢复后仍需走完整 WMS 首帧流程，实际节省可能不如预期。

### 3.2 ContentProvider 引用计数双边不一致

快照中已有 CP 实例，但 AMS 侧没有注册：

- 其他进程请求同一 CP → AMS 在 `mProviderMap` 中找不到 → 尝试启动新进程或等待 → 死锁或超时
- 快照中 CP 对象的 `IContentProvider` Binder 引用可能在 restore 后指向新的 handle 编号（如果 Binder 修复改变了 handle 编号），但 AMS 侧不知道
- `stable` / `unstable` ref count 需要 AMS 和 app 双边对齐，否则 Provider 进程永远不会被回收或过早被 kill

### 3.3 FLAG_ACTIVITY_NEW_TASK 的语义问题

方案中主动调 `startActivity` 时使用了 `FLAG_ACTIVITY_NEW_TASK`：

```java
intent.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK);
```

这会导致：
- AMS 创建新的 Task 而非复用已有 Task
- Recent Apps 列表中可能出现重复条目
- 与 Launcher 期望的任务栈结构不匹配
- 如果 AMS 认为该 app 的主 Activity 已存在于某个 Task 中，`FLAG_ACTIVITY_NEW_TASK` 的行为取决于 `documentLaunchMode` 和 `taskAffinity`——可能不会创建新实例

### 3.4 项目约束：不能修改 APP

PLAN.md 明确约束：**不支持修改安卓 APP，也不能要求 APP 厂商适配**。

虽然方案中的修改在 framework 层而非 app 层，但其影响范围是**所有 app**：
- 拦截 `handleBindApplication` 跳过初始化 → 所有 app 的 Application.onCreate 被跳过
- 不同 app 对 bindApplication 各步骤的依赖程度完全不同
- 任何一个 app 的 SDK 在 onCreate 中初始化的全局状态被跳过后，该 app 可能直接崩溃
- 特别是：某些 app 在 Application.onCreate 中注册 ActivityLifecycleCallbacks、初始化 CrashReport、设置全局异常处理器——跳过这些会导致不可预测的行为

## 四、与 PLAN.md 方案的对比

| 维度 | PLAN.md（Binder 恢复 + AMS 重连） | 本方案（Binder 恢复 + AMS 伪装冷启动） |
|:-----|:-----|:-----|
| AMS 策略 | reattachApplication：重关联现有 ProcessRecord | attachApplication：让 AMS 认为是新进程 |
| bindApplication | 不重发（走 reattachApplication 路径） | 接收但选择性跳过 |
| Activity 恢复 | 对齐现有 ActivityRecord 与 app 侧实例 | 主动 startActivity 获取全新实例 |
| 旧 Activity 处理 | 不存在"旧 Activity"问题（ActivityRecord 仍在） | 需处理幽灵 Activity 和悬挂引用 |
| ContentProvider | 重新 publishContentProviders | 需手动发布但 CP 对象状态可能不一致 |
| AOSP 修改量 | ~200-400 行（reattachApplication + restore 标记） | ~500-800 行（handleBindApplication 拦截 + PID 注入 + CP 手动发布 + 旧 Activity 清理） |
| 回归风险 | 中（新接口，不影响正常启动路径） | 高（修改核心启动路径，影响所有 app） |

## 五、结论

加上 Binder 修复前置条件后，本方案仍然不可行。核心原因是：

1. **attachApplication 不接受未知 PID** → 需要预先在 AMS 中注入 ProcessRecord（等同于 PLAN.md 的 reattachApplication 前置步骤）
2. **bindApplication 不可简单跳过** → ContentProvider 发布超时会杀进程；核心初始化步骤是后续 Activity 操作的前置条件
3. **主动 startActivity 创建新 Activity 而非恢复旧 Activity** → 旧 Activity 变幽灵，悬挂引用导致崩溃
4. **死亡通知时序竞争** → restore 过程中旧 Binder 代理的死亡通知可能触发级联清理

当把这些问题逐一修复后，本方案实质上退化为 PLAN.md 方案的一个变体：

```
Binder 修复（PLAN.md Phase 1）
  → AMS 侧预先注册 ProcessRecord（≈ reattachApplication 前置步骤）
  → 不走标准 bindApplication，走轻量重连路径（≈ reattachApplication）
  → 不主动 startActivity，对齐现有 ActivityRecord（≈ AMS-Reconnection Step 3）
  → 暂时忽略死亡通知（≈ restore 标记机制）
```

**最终建议**：放弃"伪装冷启动"思路，采用 PLAN.md 的 AMS 重连方案。两个方案在修复 Binder 后的实质性差异很小，但 PLAN.md 的设计更符合 AMS 的内部契约，修改范围更小，回归风险更低。

如果确实想探索"让 app 不走完整 bindApplication"的优化空间，建议在 PLAN.md 框架下做增量实验：先用 reattachApplication 跑通完整链路，再逐步识别 bindApplication 中哪些步骤可以在快照恢复场景下安全跳过。这样每一步都是可验证的，不会一次性引入多个不可控变量。
